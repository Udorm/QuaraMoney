# Plan: Budget & Savings (Plan tab) rework
_Locked via grill — by Claude + Udorm · Rev 6 (post Codex Round 5, final)_

## Goal

Rebuild the Plan tab's two features — budgets and savings goals — into an intuitive, best-practice flow while fixing every confirmed logic defect. Budgets become **standing rules with calendar-aligned, fully derived periods** (nothing mutates on launch); savings goals become **a direction-aware transaction ledger with derived completion**. The Plan tab becomes a **unified overview** (hero + budget cards + savings cards + insights, summary→detail like Pro Analytics). All data-model changes are **additive-only**: deprecated fields stay in place, no destructive SwiftData or Supabase migrations, existing user data keeps working. Dead machinery (rollover service, percent-of-income math, auto-contribute decoration, needs/wants/savings picker, orphaned code) is removed or revived deliberately.

## Confirmed defects this plan fixes

- Savings goals can never complete: `checkCompletion()`/`addContribution()`/`withdraw()` are never called anywhere; `currentAmount` is dead; the "Completed goals" section is unreachable.
- Rollover currency bug: `BudgetRolloverService` computes `unused = effectiveLimit (budget currency) − spent (preferred currency)`.
- Percent-of-income budgets store `amountLimit = 0`, so rollover unused is always 0 and multiple screens need a second income pass to resolve limits.
- Multi-period lapse: a recurring budget idle N periods advances only one period per launch.
- Monthly budgets run creation-date → +1 month (Jul 18–Aug 18), not calendar months.
- **Custom budgets silently lose their final day**: the end DatePicker stores midnight of the chosen day and every consumer filters `date < end`, so transactions on the selected end date never count.
- Auto-contribute is stored/synced/displayed but no service ever executes it.
- Goal-tagged transfers always count positive — withdrawals don't exist.
- Spending definition inconsistencies: rollover excludes archived-wallet transactions, list calculator doesn't; tombstoned categories aren't filtered out of `trackedCategoryIds`; `SavingsGoal.linkedTransactions` sums soft-deleted rows.
- `FilteredTransactionsViewModel` hard-deletes transactions directly, bypassing `SoftDeleteService` — no tombstone means sync can resurrect the row (pre-existing bug, fixed here because goal history drills into that screen).
- `BudgetSuggestionService` (403 lines) fully orphaned; `budgetCategoryType` picker does nothing; dev-note comments shipped in `AddBudgetView`.
- `BudgetNotificationService` has a known `loadNotifications()` overwrite race; alert state (`lastAlertThreshold`) has no per-period identity.

## Approach

1. **Branch.** Confirm `main` is pushed, branch `feat/plan-rework` from it. All work lands as one PR via the usual git workflow. No code before user sign-off on this plan.

2. **Period math (derived, calendar-aligned).**
   - `BudgetPeriodType` already has calendar-aligned `periodStart(containing:)` for weekly/monthly/quarterly/yearly — budgets just never used it. Add `currentPeriodRange(containing:calendar:)` = `periodStart(containing:)` + `dateRange(from:)`, `[start, end)` convention everywhere.
   - **Budget taxonomy:** standard-period budgets (weekly/monthly/quarterly/yearly) are always standing rules — perpetually active, current period derived from today. `.custom` budgets are one-offs with an explicit window and derived `isUpcoming`/`isActive`/`isEnded` states. `isRecurring` stops driving behavior (deprecated in place).
   - **Custom end date is inclusive-by-user-intent:** the stored `customEndDate` (midnight of the picked day) is interpreted at read time as `endExclusive = startOfDay(customEndDate) + 1 day`. No data rewrite; this also retro-fixes the existing final-day bug for old budgets.
   - **Weekly periods are device-independent**: each weekly budget persists `weekStartDay` (additive column), stamped at creation from the creating device's `Calendar.current.firstWeekday` and stamped onto existing weekly budgets by migration — every device then derives identical weekly ranges and identical alert period keys (a floating `firstWeekday` would give two devices different "this week"s and dueling dedupe keys). Non-weekly budgets ignore it.
   - `.biweekly` removed from creation UI; the enum case remains readable (old data, old clients) but normalization (step 6) converts encountered rows to weekly.
   - Recurring budgets count the **whole current calendar period**, including transactions dated before the budget was created (YNAB semantics). `startDate` is kept as "budget began" metadata only.

3. **One spending definition, one calculator.**
   - `BudgetCalculator` becomes the single authority: expense transactions with `deletedAt == nil`, `event == nil`, `!excludeFromReports`, matching tracked categories; **archived-wallet transactions count** (spending happened regardless of later archival).
   - **Explicit target kind:** add `targetKindRaw: String?` (`total` / `categories`) so "budget everything" and "budget whose categories all disappeared" stop sharing a representation. **The getter is nil-lossless**: `nil` derives the kind from relationship presence (any `categories`/legacy `category` ⇒ `categories`; else `total`) — identical to the backfill rule — so a row pulled from an old client behaves correctly *before* any normalization pass writes to it (`alertMode`'s getter derives from the legacy toggles the same way). Backfill just materializes the derivation. After that: `targetKind == .categories` with zero live (non-tombstoned) tracked categories renders a **needs-attention** state, never a silent total budget. (`trackedCategoryIds` always filters `deletedAt == nil`; legacy single `category` is read-merged, never written.)
   - Progress is computed **in the budget's own currency**. Honest FX statement: transactions already in the budget's currency compare exactly; cross-currency transactions convert at **live rates** (there is no per-transaction base-currency snapshot in the schema, and adding one is out of scope), so cross-currency progress can shift slightly on rate refresh. Displayed amounts convert to preferred currency as today. Single-pass batch APIs (`spendingByBudget`) retained and adapted.

4. **Kill dead machinery; revive the suggestion engine.**
   - Delete `BudgetRolloverService` and its startup-maintenance step (the guarded maintenance gate keeps only category work — simplification, no gate semantics change). Rollover UI, `rolloverExcess`/`rolloverAmount` reads, percent-of-income UI/math, needs/wants/savings picker, auto-contribute section, and the goal "Mark active" button all go. Purge the shipped dev-note comments.
   - Rework `BudgetSuggestionService` → `BudgetSuggestionEngine`: per-category (and total) suggested amount over the last **3 completed calendar periods of the selected period type as zero-filled buckets** (a no-spend period counts as 0; a category with no usage at all ⇒ no-data confidence, no suggestion), +10% buffer, confidence tiers. **FX is failure-aware**: amounts whose rate is missing (would silently fall back 1:1 today) downgrade confidence and are flagged, not silently converted. **Computed in a background `ModelContext` returning `Sendable` snapshot values** (the established `WalletBalanceStore`/SuggestionSnapshot pattern — three *yearly* buckets or a total-budget scan is not main-actor-cheap; live models never cross actors). **Custom one-offs get no historical suggestion** — there is no repeating unit to average; the amount field is simply empty.

5. **Model changes (additive only).**
   - `Budget`: add `targetKindRaw: String?` (step 3), `alertModeRaw: String?` (`off` / `nearing` / `overOnly` / `nearingOver`; nil ⇒ exact mapping from legacy toggles, step 7) and `lastAlertPeriodKey: String?` + keep `lastAlertThreshold` for per-period alert dedupe. Deprecated in place (kept, unread by new logic): `month`, `year`, `isRecurring`, `rolloverExcess`, `rolloverAmount`, `amountTypeData`, `budgetCategoryTypeRaw`, `alertAt50`, `alertOnProjectedOverspend`.
   - `SavingsGoal`: deprecated in place: `autoContributeEnabled/Amount/PeriodRaw`, `currentAmount`. If a synced legacy `currentAmount > 0` exists it is displayed as a read-only "starting balance" line and included in totals — never writable. Migration stamps a new additive `startingBalanceCurrencyCode` (= goal currency at migration time) and the balance is **always interpreted through that stamp**, so a later currency change — from any client version — can never re-denominate it. With denomination pinned, goal currency stays freely editable; the edit sheet offers to convert the target amount at the live rate.
   - `Transaction`: add `savingsIsWithdrawal: Bool = false` — **the only new savings column**. The ledger amount is *derived from transfer facts already on the row*, not snapshotted: contribution ⇒ destination-side value, withdrawal ⇒ source-side value, resolved by a **single shared transfer-side amount resolver used by both wallet-balance math and the savings ledger** (rate chain: `storedRate` → legacy `exchangeRate` → fallback rates — exactly what `Wallet+Extensions` does today, extracted so the two can never diverge; all three paths tested). No save-time FX call, nothing an old client can half-update: old clients that edit amount/rate keep wallet math and goal ledger consistent by construction, and old-client-created goal transfers default to `false` = contribution, which matches their UI's only capability. The Boolean is **authoritative** for sign — no wallet-direction inference anywhere. Converting wallet-side value to the goal's display currency happens live and failure-aware (missing rate ⇒ row shown unconverted with a hint, excluded totals flagged — same policy as suggestions).
   - Supabase: additive columns `budgets.target_kind`, `budgets.alert_mode`, `budgets.last_alert_period_key`, `budgets.week_start_day`, `savings_goals.starting_balance_currency_code`, `transactions.savings_is_withdrawal` — new migration file, mirrored into `schema.sql`, applied to the live project **before** the client ships. Deprecated columns keep syncing unchanged (old clients unaffected). **Verification requirement before ship**: an old-client-shaped transaction upsert (payload without `savings_is_withdrawal`) must be replayed against the live schema to confirm PostgREST leaves the existing column value untouched (expected: partial-column upsert never resets omitted columns; if that expectation ever fails, withdrawals cannot ship without gating — this is checked, not assumed).

6. **Two distinct maintenance passes** (both inside the existing account-settled, generation-guarded startup maintenance; both set `needsSync` + `updatedAt` on touched rows):
   - **One-time migration, marker-gated**: per-owner marker — key is the owner UUID, or an explicit `"local"` sentinel for stores never claimed by a cloud account (startup maintenance runs for sync-disabled users too) — plus version, same completion-flag pattern as the perf pass's maintenance guard, committed **only after the guarded save succeeds**. Account adoption changes the identity, so the migration re-evaluates for the new owner; it is idempotent, making the re-run safe (tested across the adoption boundary). **Failure re-arms for real**: neither the durable marker *nor the in-session `accountMaintenanceCompleted` flag* is set on failure, so the existing foreground/settle triggers retry in the same session — the current gate's fire-and-forget `.failed` path is explicitly upgraded, not assumed. Ordered contents: **(a) percent-of-income freeze first, while each budget's original period type is still readable** (stored percentage × last *completed* period of the *original* type's income, in the budget's currency; income 0 ⇒ limit 0 + "set your limit" needs-attention); **(b) then legacy one-off conversion** — capture the original `[start, end)` window *before* touching `periodTypeRaw`, convert non-custom non-recurring budgets (including biweekly one-offs, whose full 14-day window must survive) to `.custom` with `customEndDate` = final *inclusive* day of that window; (c) `targetKindRaw` + `alertModeRaw` materialization, `weekStartDay` stamp for existing weekly budgets, and `startingBalanceCurrencyCode` stamp for goals with legacy balances; (d) savings completion backfill via the reconciler (step 8) so already-over-target goals complete immediately.
   - **Always-run normalization, idempotent and cheap** (a no-op scan when there's nothing to heal), so rows written later by old clients converge: percent freeze (same rule as above), *recurring* biweekly → weekly (semantic change, release-noted), **non-custom non-recurring → `.custom` one-off conversion (same rule and same freeze-before-convert ordering as the migration — an old client can create a one-off monthly budget at any time, and without this it would masquerade as a standing rule on new clients)**, nil raw-field materialization. Runs at every startup-maintenance pass **and after each sync pull applies budget changes** (on the existing post-apply main-actor hop) — a percent budget pulled from an old client is healed immediately, and thanks to the nil-lossless getters (step 3) even the pull-to-heal window behaves correctly.
   - No other date rewriting: calendar alignment is pure derivation. Release-note that current-period numbers change on update day.

7. **Notifications rebuilt.**
   - `BudgetNotificationService` re-implemented with a single load path (fixing the overwrite race). **Evaluation is app-level, not view-level**: debounced `.dataDidUpdate` while the app is active + a sweep on foregrounding — a transaction added from any screen can fire a threshold alert; Plan-tab visibility is irrelevant.
   - **Delivery-idempotent**: notification request IDs are deterministic (`budgetAlert_<budgetID>_<periodKey>_<threshold>`), so a crash between scheduling and state-save replaces rather than duplicates; dedupe state (`lastAlertPeriodKey` + `lastAlertThreshold`) is saved only after the add succeeds, and scheduling failures are logged and retried on the next evaluation. **Period keys are calendar-component labels, not instants** — `2026-W29` / `2026-07` / `2026-Q3` / `2026` (custom: the budget ID alone, one period) — so the key format is identical across devices and time zones. Absolute period-boundary *instants* remain device-local, matching how every other screen in the app (analytics, Home, transaction grouping) already interprets dates; a user running devices in two time zones simultaneously can see boundary transactions land in adjacent periods, which is the app's existing, accepted date semantics, not a regression.
   - Thresholds come from `alertMode`, which gains a fourth value for lossless migration: `off` / `nearing` (80%) / `overOnly` (100%) / `nearingOver` — still one simple picker. Legacy mapping is exact for 80/100 combinations (both off → off, 80-only → nearing, 100-only → overOnly, both → nearingOver); the 50% and projected-overspend toggles are dropped and release-noted. Alert content localized via L10n (English + Khmer), addressing the English-notifications audit finding for budget alerts.

8. **Savings ledger semantics (one predicate, one reconciler).**
   - **Eligibility predicate** (single shared implementation): `deletedAt == nil && type == .transfer && savingsGoal == goal`. Sign comes from `savingsIsWithdrawal` alone (authoritative; legacy and old-client rows default to contribution, which is historically exact). Amount per step 5's derivation from transfer facts.
   - **All transaction deletion goes through `SoftDeleteService`**: `FilteredTransactionsViewModel`'s direct hard-delete is replaced (fixing the pre-existing tombstone-bypass sync bug), giving deletion a single choke point the reconciler can hook.
   - **`SavingsGoalReconciler`** is the only writer of `isCompleted`/`completedDate`: recompute total, set `isCompleted = total ≥ target` (`completedDate` stamped when crossing up, cleared when crossing down). Invoked from every mutation path: transaction save/delete choke points (`AddTransactionViewModel`, `SoftDeleteService`), goal edits (re-tagging reconciles **both** the old and new goal), **post-sync-pull apply** (on the existing `didApplyRemoteChanges` main-actor hop), **exchange-rate refreshes** (`ratesDidChange` — cross-currency ledgers can cross the target on a rate move alone), and migration backfill. **UI state is belt-and-braces derived**: active/completed list membership and the detail header read the live derived total, not the stored flag, so even an unreconciled instant renders correctly — the stored flag exists for cloud/filter compatibility.
   - **Pull- and rate-triggered reconciliation is local-only** — it corrects `isCompleted` in the store but must **not** set `needsSync`. Because `SyncMutationTracker` stamps `needsSync` on any observed save, these reconciliation saves run inside the engine's synchronous `withSyncWriteGuard` span (never across an `await`, per the tracker's contract), with a test asserting the goal row stays clean. This prevents completion flapping when one device's goal row arrives before its contributing transaction (the receiver would otherwise derive `false` and push it back over the sender's `true`); since completion is derived, the cloud flag is just a cache and local derivation always wins. Local-mutation-triggered reconciliation marks `needsSync` as normal. Push order stays goals-before-transactions — transactions carry an FK to goals, so reordering would break referential integrity on insert.
   - **Over-withdrawal defined**: the raw ledger may go negative through edits/sync; display and progress floor at 0 with no crash states; the Withdraw sheet validates amount ≤ current saved as a best-effort guard, not an invariant.

9. **UI rebuild** (Liquid Glass, `.appFont`, CornerRadius tokens, `AppEmptyStateView`, `.contentShape(Rectangle())` on tappable rows, summary→detail like Pro Analytics):
   - **`PlanOverviewView`** replaces the `BudgetTabView` segmented shell. **The hero is non-additive** (mixed-period budgets cannot be summed without double counting): it shows this month's total spending (existing analytics definition) plus budget health — "x of y budgets on track" and the highest-risk budget — and, only when a monthly **total** budget exists, that budget's own left-to-spend as the headline. Quick-create blocks creating a second standing total budget with the same period type; if duplicates arrive via sync anyway, the hero picks the oldest by `createdAt` (deterministic) and the extras surface as a needs-attention duplicate state. Budget summary cards, savings goal cards, and an insights card (over/under pace, top category, from the suggestion engine) each drill into their list/detail. Single `+` menu (New budget / New goal). Segmented control and per-section FAB die; search moves into the drill-in lists.
   - **Budget list**: standing budgets grouped as "Your budgets"; `.custom` one-offs grouped by upcoming/active/ended; simplified filter (menu, not the wallet-shaped `FilterSheetView`). Needs-attention budgets surface at the top with their fix action.
   - **Budget detail**: donut + summary retained and refreshed; **period navigator (‹ ›)** to view past calendar periods — labeled honestly as *historical spending evaluated against the current rule* (limit, categories, and currency are today's; the rule isn't versioned). **Hidden for `.custom` one-offs**, which have no adjacent periods — they show their single inclusive window. Alerts row reflects `alertMode`; transactions drill-in unchanged.
   - **Quick-create budget** (replaces `AddBudgetView` form): pick target (category chips or Total) → amount pre-filled by suggestion with "you averaged X/month" context and confidence → Save. Defaults: monthly standing rule, alerts Nearing + Over, preferred currency. An Options disclosure holds period type (weekly/monthly/quarterly/yearly/custom one-off with dates), name, currency, alert mode. **Suggestions are period-type-aware**: changing the period type recomputes the three zero-filled buckets in that unit and updates the "you averaged X/week|month|…" copy — a monthly average is never silently saved as a weekly limit. `EditBudgetView` reuses the same components.
   - **Savings**: goal detail gets direction-aware ledger rows, an explicit **Withdraw** action (transfer sheet with `savingsIsWithdrawal = true`), derived completion with celebration on crossing; the reconciler keeps the stored flag honest. `AddSavingsGoalView` stays template-first, loses the auto-contribute section. List keeps active/completed split, now actually functional.
   - `BudgetInsightsView` content folds into the overview insights card + budget detail; its `MoreView` entry is removed.

10. **Localization**: all new strings added to both `en.lproj`/`km.lproj` and `String+Localization.swift` regenerated (never hand-edited).

11. **Tests** (in-memory `TestModelContainer` pattern): calendar period math (month lengths, leap Feb, quarter/year boundaries, firstWeekday variants, `[start,end)` boundary transactions, **custom final-day inclusivity**); unified spending definition incl. archived wallets + tombstoned categories + `targetKind` backfill + needs-attention rule; budget-currency progress; migration transforms (percent→fixed **using original period type before one-off conversion**, recurring-biweekly→weekly, one-off conversion preserving full original windows incl. biweekly's 14 days, alertMode/targetKind exact materialization + **nil-getter derivation parity**, marker durability + failure re-arm in-session, normalization idempotent re-run healing old-client writes incl. post-pull); savings amount derivation via the shared transfer-side resolver (same-currency, `storedRate`, legacy `exchangeRate`, missing-rate fallback, withdrawal source-side), reconciler from every hook incl. `SoftDeleteService`-routed deletes, re-tag old+new goal, rate-change, sync-pull local-only (**goal row stays `needsSync`-clean under the write guard**) and **seeded legacy stores** (completion backfill + reactivation), over-withdrawal floor; alert per-period dedupe + deterministic-ID idempotency + all four alertMode mappings; suggestion zero-filled buckets per selected period type + FX-failure confidence downgrade + background-context Sendable snapshots + no-suggestion-for-custom; weekly `weekStartDay` parity (two calendars, same ranges + alert keys) + component-based period-key format stability; normalization converts old-client-created one-offs (post-migration standard-period `isRecurring == false` rows never render as standing rules); marker identity across local→account adoption; starting-balance denomination pinned through goal-currency change; navigator hidden for custom one-offs; hero total-budget dedupe (oldest-wins).

12. **Verification**: full build + test suite; drive the real flows (create budget via suggestion, cross a threshold from the Home quick-add path, view past period, contribute + withdraw + complete + reactivate a goal, migration from a seeded legacy store) before the PR.

## Key decisions & tradeoffs

- **Derived periods over stored instances** — kills the advance-on-launch bug class; tradeoff: no persisted per-period rule history, so past periods are shown against the *current* rule (labeled as such in the UI).
- **Whole-calendar first period** — a budget created Jul 18 counts all of July; simpler and matches user expectation, at the cost of "surprising" spend already counted on day one.
- **Rollover dropped, not fixed** — removes the last stateful mutation; users who had it enabled lose the behavior (release note).
- **Percent-of-income → creation-time suggestion** — one-pass math, predictable limits; irregular-income users lose live-scaling limits.
- **Savings = ledger only, derived from transfer facts** — one source of truth; amounts derive from each row's own `amount`/`storedRate`/direction flag (no snapshot columns, nothing an old client can half-update); legacy `currentAmount` demoted to a read-only starting balance.
- **Auto-contribute dropped** rather than extending the recurring engine to transfers (logged as future work).
- **Progress in budget currency, live FX for cross-currency rows** — no snapshot infrastructure added; same-currency budgets (the dominant case) are exactly stable, cross-currency ones honestly aren't.
- **Archived-wallet spending counts** — resolves the calculator/rollover inconsistency in favor of "spending happened".
- **No minimum-client-version sync gate** — the app has no version-gating infrastructure and building one is out of scope; instead normalization is idempotent (old-client writes heal on next pass) and old-client damage is bounded to display-only skew (see risks).
- **Additive-only schema** — deprecated fields remain in models, DTOs, and cloud columns; ugly but zero migration risk and old clients keep working.

## Risks / open questions

- **Mixed-version devices** (stated fully): old clients keep running the rollover service and can mutate `startDate`/`rolloverAmount` (new client ignores both); they can create percent/biweekly budgets (healed by the always-run normalization pass on a new client, which may visibly change that budget); they show goal-tagged withdrawals as positive contributions (display-only skew — old clients never persist derived totals, and because ledger amounts derive from ordinary transfer fields, an old client editing a transfer keeps wallet math and goal ledger consistent); they won't see needs-attention states. **Residual accepted risk**: an old client re-tagging a *withdrawal* row to a different goal carries the hidden `savings_is_withdrawal = true` with it — the row still reduces (not raises) the new goal's total on new clients; gating old-client edits is infeasible without version infrastructure (out of scope), the scenario requires the same user editing on an outdated second device, and the direction is visible and one-tap-fixable in the new UI. No data corruption path identified; period displays differ between versions until both update.
- **Update-day discontinuity**: calendar snapping + percent freezing changes displayed numbers with no data loss; needs a release note / one-time in-app notice.
- **Percent→fixed freeze quality**: last-completed-period income may be unrepresentative; mitigated by the needs-attention prompt.
- **Supabase columns must be applied to prod before the client ships** (established workflow, but a hard ordering constraint).
- Suggestion quality for sparse Cambodian-market data (few transactions, KHR/USD mix) — zero-filled buckets + FX-failure downgrade + no-data tier must degrade copy gracefully to "no suggestion".
- `targetKind` backfill can misclassify a category budget that had *already* lost every category before migration (indistinguishable from total) — it backfills as `total`; rare, documented, user-fixable by editing the budget.

## Out of scope

- Recurring transfers / auto-contribute execution (future work, logged).
- 50/30/20 or needs/wants/savings templates.
- Persisted per-period budget history (rule versioning).
- Per-transaction base-currency FX snapshots for budget math.
- Minimum-client-version sync gating.
- Shared/group budgets, event-ledger interaction, debts.
- Destructive cleanup of deprecated fields/columns.
- Any change to sync engine architecture beyond additive DTO columns.
