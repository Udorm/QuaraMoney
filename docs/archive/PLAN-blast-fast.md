# Plan: Whole-app performance pass ("blast-fast")
_Locked via grill вЂ” by Claude + Udorm В· Rev 7 (post Codex Round 6, user-extended вЂ” all findings folded in)_

## Goal

Make QuaraMoney feel instant вЂ” cold launch first, then every surface вЂ” via a **code-only audit-and-fix pass** (no runtime instrumentation, no Instruments traces): trace hot paths by reading code, fix everything provably wasteful, and ship one branch + PR whose description doubles as the findings list. The architecture stays exactly as-is (SwiftUI + SwiftData + MVVM, singleton managers, `.dataDidUpdate` refresh pattern, Supabase sync). Acceptance philosophy: **first frame fast, fill in async** вЂ” screens appear and respond immediately; deferred values settle a beat later, but **never render a fabricated number as final** (see the progressive-loading contract below). This builds on the landed 2026-07 perf overhaul, targeting what it missed or deferred.

## Approach

1. **Git hygiene first:** local `main` (`23741a7`, a UI-fix commit) is one commit ahead of `origin/main`. Push it to origin before branching вЂ” direct commits to `main` are an established pattern in this repo's *first-parent* history (`36f34ed`, `56fc55a`, `2b6fc2f` all landed directly between PR merges); if branch protection rejects the push, land it through its own PR instead. Then branch `perf/blast-fast-pass` from the updated remote base.
2. **Splash: hard-capped в‰¤ 0.5 s total, including fade.** Replace the fixed 1.5 s hold + 0.3 s fade with ~0.3 s hold + 0.2 s fade. **No content-readiness signal** вЂ” the content is already mounted beneath the splash at opacity 0, so there is no "not ready" state to wait for; the readiness-gating idea from Rev 1 is dropped as needless complexity. Keep `_hasShownSplash` once-per-process behavior and fire `transferPendingShortcutToRouter()` on splash completion exactly as today.
3. **App-lock becomes synchronous.** `securityManager.isAppLocked = isAppLockEnabled` currently runs in a `.task` (post-first-render); with a short splash that's a visible-content race for a finance app. Move the cold-launch lock decision to scene construction (App `init` / initial `@State`) so the lock overlay is present from the very first frame. This is a hard precondition for step 2 shipping.
4. **Explicit startup phasing** (replaces "verify tasks don't land on the first-frame window"):
   - **Critical phase (at construction / first frame):** ModelContainer, `SyncMutationTracker.start` + `enableAutoSync` (must not miss early user edits), app-lock state (step 3), language/font environment. `authManager.start()` keeps its current scene-`.task` position вЂ” its phase is now explicit, not accidental.
   - **`UIFont.setupAppAppearance()` moves to the critical phase** (App `init`): it configures appearance proxies for nav/tab bars already mounted beneath the splash вЂ” with a 1.5 s splash the late application was invisible, but a 0.5 s splash would expose a font pop. Only the **font-cache prewarm** stays deferred: kicked off from the splash **fade-completion** callback plus a main-actor yield вЂ” not from the callback that *starts* the fade, so it never competes with the 0.2 s transition or the first visible frame.
   - **Deferred, database setup** (`setupServices()`: recurring reminder reschedule, budget rollovers, category stamp/seed, `fetchRates()`): requires **both** splash fully removed **and** onboarding completed. It stays anchored to `ContentView` (which only exists post-onboarding, exactly as today) and additionally awaits the splash-removed signal вЂ” preserving the current invariant that seeding/rollovers never run during onboarding.
   - **`BudgetNotificationService` loading: timing untouched.** Its existing 2 s-settle task is already off the first-frame path, and its `loadNotifications()` has a known overwrite race (also invocable from `NotificationCenterView`) вЂ” this pass neither re-anchors nor fixes it; flagged in PR only.
   - **Rollover/ownership race hardening:** the current `ownedAtStart` snapshot is taken *after* `BudgetRolloverService` has already committed internally, so a post-hoc check can't undo it. Refactor the rollover service to **return its changes unsaved** (or accept a caller-supplied commit closure) so the single startup save happens under the same snapshotв†’checkв†’save/rollback guard as the category work. **Rollover notifications become value-type payloads returned with the changes and are scheduled only after the guarded save succeeds** вЂ” a rolled-back rollover must never deliver a "Budget Rolled Over" notification.
   - **Account-settled gating вЂ” safe terminal states only:** startup DB maintenance (rollovers, stamp/seed, **and the recurring-reminder rebuild** вЂ” `rescheduleAll` wipes and re-schedules from whatever store is present, so it must not run against a pre-reconciliation store) runs only when ownership cannot be about to change: sync disabled; signed out; or signed in with the full post-sign-in pipeline complete вЂ” `reconcileAccountIfNeeded` done, first-sign-in conflict check resolved, **and the initial sync idle/completed** (the pipeline's `syncIfOperational` pulls and mutates budgets/categories; maintenance must not race it). While the state is *unknown* (session restoring, conflict check or initial sync in flight, `.conflict`/`.checkFailed` pending), maintenance does not run вЂ” it is **skipped and re-armed** to retry when the state next settles (auth-state change or next launch), never run anyway. Rollovers and reminder rebuilds are non-urgent; skipping a launch is free, committing under unknown ownership is not. Maintenance reruns after any reconciliation/conflict resolution so reminders always reflect the adopted account. **The reminder rebuild is a post-save effect:** `rescheduleAll` deletes and re-adds requests across multiple awaits and cannot be rolled back, so it runs only *after* the guarded save and a final generation check вЂ” and whenever a generation is invalidated mid-flight, the recurring-request prefix is cleared so no wrong-account notification survives cancellation.
   - **Gate mechanics:** the once-only `didRunSetupServices` flag splits in two вЂ” one-shot global setup stays once-only, while account maintenance gets its own completion flag set **only after its guarded save succeeds** (otherwise the skip/re-arm would no-op against the existing guard). **The entire maintenance attempt вЂ” not just the settlement wait вЂ” is bound to user ID + auth generation**, invalidated on every auth transition, with revalidation after every suspension point: before context creation, immediately before save, and before any post-save effect. A sign-out or account switch mid-attempt can never let stale work resume against the new account's store. The gating decision itself is a **pure function** (account/auth/conflict/sync state в†’ run/skip), unit-testable without network.
   - **Ownership snapshot is identity, not a Boolean:** the existing guard's `isLocalStoreAccountOwned` comparison passes an Aв†’B account switch unchanged (ownedв†’owned). The guard this plan builds snapshots and compares the exact `localOwnerUUID` + auth generation, catching owner *changes*, not just owned/unowned flips. The residual checkв†’save instant remains as a documented known limit (a full ownership/commit coordinator is an architecture change this plan forbids; flagged in PR as a follow-up) вЂ” but with identity comparison its blast radius is a re-check race, not a wrong-account write silently accepted.
   - **Rollover rates freshness:** startup currently snapshots `CurrencyManager.rates` before rollovers run, while `fetchRates()` fires later вЂ” a rollover persisted with stale rates can't be un-saved by the later signal. The maintenance path awaits the (24 h-throttled, usually instant-cache) rate refresh before capturing rates for rollover, falling back to the cached table when the refresh fails or times out.
5. **Scene-invalidation audit, mapped not assumed.** First build a consumer/property map of the remaining `ObservableObject` singletons (who observes what, which properties churn, which `$`-bindings and `@AppStorage` interactions exist). Then:
   - **Migrate:** `SecurityManager` only (root-observed, multi-property: lock state, biometric config; `@Published` writes re-evaluate the whole scene while only the lock overlay cares) вЂ” following the established `CurrencyManager`/`ThemeManager` `@Observable` pattern, converting `$` bindings to `@Bindable` explicitly.
   - **Do not migrate:** `ErrorService` (one rarely-written property that the root *must* observe to present the alert вЂ” `@Observable` can't narrow that; no provable win, so it fails this pass's own inspection bar), `NotificationManager` (its daily-reminder scheduling calls `removeAllPendingNotificationRequests`, nuking recurring + budget notifications вЂ” a pre-existing cross-feature bug; flag in PR, don't perturb), `BudgetNotificationService` (load race, see step 4; flag in PR), `SupabaseAuthManager` (lives in the read-only Supabase directory).
6. **Whole-app waste sweep** (every feature area *except* `QuaraMoney/Supabase/`):
   - **hot-path** per-call formatter construction (row rendering, per-frame paths) вЂ” one-per-operation formatters in exports/imports are fine and stay local
   - fetch-all-then-filter where a `#Predicate`/`fetchLimit` would do (`EventDetailViewV2` fetches the global ledger and filters per-event in memory вЂ” named target). **Query-change rules:** any added limit requires a deterministic sort descriptor first; display queries ("recent N") stay separate from aggregate/total queries; material query-semantics changes get a targeted unit test.
   - unbatched or visibility-ignoring `.dataDidUpdate` handlers (must use the gated + generation-checked pattern)
   - heavy computed properties in `body`; `ForEach` identity and row-body cost in long lists; redundant `.id(...)` rebuilds; `onAppear` work storms
   - **all unstructured concurrency** вЂ” not just `Task.detached`: any `Task {}` or suspension point capturing SwiftData models across actor/async boundaries gets value snapshots or `PersistentIdentifier` instead (e.g. `BudgetRolloverService` notification closures capture `Budget`).
7. **Progressive-loading contract:** wherever this pass defers a money/analytic value off the first frame, the UI must show a placeholder (`.redacted`), a spinner, or a value explicitly marked stale/last-known вЂ” never an initialized-to-zero figure presented as real. **Home is a named target:** before `hasLoadedOnce`, `HomeViewModel`'s zero-initialized totals flow into the summary cards as real values вЂ” redact/withhold until the first result lands. Audit `AnalysisViewModel`-style zero-defaults on every other screen this pass touches.
8. **Rates-staleness fix, inventory-first:** `fetchRates()` mutates `CurrencyManager.rates` silently; deferring it further (step 4) widens the stale window. **First** inventory every rate-dependent consumer (Home, wallet detail, filtered transactions, debts, recurring progress, budgets, summary charts, analysis/pro processors, `WalletBalanceStore`) and classify each: reads-at-render (needs nothing) vs snapshot-holder (needs the signal). **Then** add a dedicated rates-changed signal (notification, mirroring `.dataDidUpdate` conventions) posted after a successful fetch. **Every snapshot-holder handles the signal via the established visibility-gate pattern** вЂ” visible now в†’ refresh now; not visible в†’ mark dirty and refresh on next appearance. That pattern subsumes on-appearance refresh, so a screen that is on-screen when rates arrive updates immediately, and no consumer that displays money ever ships knowingly stale. Consumers lacking a gate today adopt the same pattern (it's the codebase's standard), not new machinery.
9. **Heavy-screen deep dives:** ProAnalytics processor (single-pass? re-processing on non-material changes?), Analysis tab, Budget/Plan tab, Add Transaction open latency (verify nothing synchronous crept back in front of the async suggestion snapshot), Event detail V2 settlement recompute frequency.
10. **Supabase directory: strictly read-only.** All of `QuaraMoney/Supabase/` (engine, tracker, realtime, profile sync, **and** `SupabaseAuthManager`) is flag-only вЂ” findings go in the PR description as flagged-not-fixed. Only call-site *scheduling* outside that directory may change. (The Supabase client is already lazy and the auth init already empty вЂ” no launch win there anyway.)
11. **Verify:**
    - `xcodebuild build` + full unit suite green; new targeted tests for any changed query semantics (step 6) pass.
    - **Startup-race unit tests:** the rollover refactor (step 4) makes the guarded commit testable вЂ” add a test that flips ownership state between snapshot and commit on an in-memory container and asserts rollback (no stale write survives). The account-gating **pure function** gets network-free tests over every state: sync-disabled в†’ run; signed-out в†’ run; signed-in fully settled (incl. initial sync idle) в†’ run; session-restoring / conflict-pending / `.checkFailed` / initial-sync-in-flight / timeout в†’ skip (and the re-arm path fires on settle); a stale settlement completion from a previous auth generation is ignored. **Event-sequence tests cover post-settlement interleaving:** an Aв†’B account switch after settlement but before save (identity comparison catches the ownedв†’owned change; no stale write survives) and before post-save effects (no reminder/notification from the stale generation is scheduled). Tests the decision logic and guard directly; no new injectable "startup coordinator" is built for this.
    - Simulator smoke matrix, fresh-install split into its two real cases: (a) fresh install в†’ complete onboarding в†’ starter wallet exists, **no** wallet-creation sheet; (b) already-onboarded app with an empty store в†’ mandatory wallet-creation sheet appears. Plus: cold launch with app-lock ON (lock visible from first frame) and OFF; all four tabs; both Add-Transaction variants (classic + compact); a language switch (ENв†”KM) mid-session; a debug-seeded larger dataset for list-scrolling sanity.
    - Signed-in/sync/conflict launch paths: exercised only as far as the simulator allows without touching cloud state; the sync engine is unmodified and the startup-race guard is unit-tested, which together cap that risk. User does the final on-device feel check.
12. **Ship:** PR to `main` with the complete fix list (grouped: launch / invalidation / waste / flagged-only) in the description.

## Key decisions & tradeoffs

- **Code-only, no measurement** (user's explicit pick): fixes must be *provably* wasteful by inspection вЂ” artificial timers, redundant/duplicated work, needless breadth. No speculative micro-optimization. Tradeoff: no before/after numbers; user validates feel on device.
- **Fix, don't document:** everything is fixed on the branch or flagged in the PR description. No audit doc.
- **Bold on launch, hands off sync:** startup reordering authorized *with* the phase model and ownership-guard hardening above; `QuaraMoney/Supabase/` (including the auth manager) is read-only.
- **First-frame-fast, honestly:** progressive fill-in is acceptable only under the placeholder/stale-marking contract вЂ” perceived speed never fabricates financial figures.
- **Splash = hard в‰¤ 0.5 s including fade** (user confirmed the brand-beat direction; the cap resolves Rev 1's unbounded "whichever is later").
- **`@Observable` migration is evidence-scoped:** `SecurityManager` only вЂ” the sole case where root-scene invalidation breadth provably exceeds what any observer needs. `ErrorService` stays: its single rare property is exactly what root must watch. The two notification services are excluded because each hides a pre-existing bug this pass must not perturb.
- **Architecture frozen:** MVVM, singletons, `.dataDidUpdate`, `BaseViewModel`, parallel Add-Transaction views all stay.

## Risks / open questions

- **No numbers:** mitigated by restricting fixes to categorical waste where the win is definitionally real.
- **Startup reordering:** the phase model + extended ownership guard + fresh-install smoke are the gates; seeding *logic* is untouched.
- **`SecurityManager` migration** touches the app-lock gate; step 3 makes lock-at-launch *stronger* (synchronous), and the smoke matrix covers lock on/off cold launch.
- **Rates signal** adds a new notification; consumers already have gated refresh paths, so the blast radius is a debounced recompute.
- Simulator smoke в‰  device reality; user does the final feel-check on hardware.

## Out of scope

- `QuaraMoney/Supabase/` changes of any kind (flag-only), schema/cloud changes, anything in `supabase/`.
- Fixing the `NotificationManager` remove-all bug and the `BudgetNotificationService` load race (pre-existing, non-perf; flagged in PR as follow-ups).
- New dependencies, architecture changes, runtime instrumentation, launch A/B measurement.
- App size, build time, memory footprint (unless incidental).
- Merging/deleting either Add-Transaction variant; re-doing the landed 2026-07 perf overhaul.
