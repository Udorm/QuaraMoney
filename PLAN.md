# Plan: Budget & Savings UI/UX redesign (Plan tab v2)
_Locked via grill — by Claude + Udorm · Rev 6 (post Codex Round 5, final)_

## Goal

Redesign every screen of the Plan tab's two features — budgets and savings goals — into a simple, native, three-level hierarchy adapted from the user's sample mock: **overview (two summary cards) → dedicated list → detail**, with **one shared form per feature for both create and edit**. This is a **UI/flow redesign**: no SwiftData model changes, no Supabase schema changes, no changes to `BudgetCalculator` period semantics, `BudgetSuggestionEngine` scoring, alert logic, or contribution mechanics (all just hardened in PR #20). The redesign *does* include a small, tested **Plan metrics layer** the new screens genuinely need — named honestly below, never buried in views. Visual language is the app's existing flat-card system (Pro Analytics `ProCard` recipe, continuous corners, `systemGroupedBackground`), with soft color tints on the two overview cards; native `insetGrouped` lists; Liquid Glass only where iOS 26 puts it natively (buttons, toolbars). All new strings localized EN + Khmer; every replaced view file is deleted.

## Approach

1. **Branch.** `feat/plan-ui-redesign` from pushed `main`. One PR via the usual git workflow. No code before user sign-off on this plan.

2. **Shared scaffolding.**
   - `PlanCard` — flat continuous-corner card (`RoundedRectangle(cornerRadius: 20, style: .continuous)`, `secondarySystemGroupedBackground` fill) matching the Pro Analytics `ProCard` recipe (which stays private to Pro — a small local copy avoids cross-feature coupling). Supports an optional tint fill for the overview cards.
   - `PlanProgressBar` — capsule track + gradient fill (recipe already in `SavingsGoalRowView`), parameterized by color; one component for overview cards, list rows, and detail stat cards. Accessibility value = percent.
   - **`PlanRefreshPolicy` — one shared invalidation contract for every Plan screen** (overview, both lists, both details): appear · visibility-gated `.dataDidUpdate` · rate refresh · preferred-currency change · language change · `NSCalendar.dayChanged` · significant-time/timezone change · scene becomes active. All refreshes run through the established generation-checked pattern (stale results dropped by token). List loaders additionally key their generation request on the **selected segment**; search text filters loaded results **in-memory only** and never triggers a fetch.

3. **Plan metrics layer (new, tested — the only new logic in this redesign).**
   Follows the repo's established loader/compute split (`WalletBalanceStore` / SuggestionSnapshot pattern): context-bound loaders fetch **value snapshots** (never models across actors), pure `Sendable` compute functions turn snapshots into screen state, all deterministic under injected date/calendar/rates.
   - **Determinate vs indeterminate — FX failure is modeled, not footnoted.** Every projection carries `isDeterminate`. The common guarantee is **"rate-checked, never 1:1 fallback"** — not one universal function: budget-side conversions use `CurrencyManager.convertOrNil` (**hardened in this PR to require both source and target rates finite and > 0** — today a zero/invalid target rate silently converts amounts to zero; tests cover zero, negative, NaN, infinite), while the savings ledger keeps the reconciler's **strict live-rate-table semantics** (both currencies must be present — no fallback-rate consultation), preserving exact parity. Any excluded amount makes the owning projection **indeterminate**: the UI shows the partial-data state, **makes no on-track / behind / completed claim** for that item, and indeterminate items are excluded from every `N of M` classification (the line reads "N of M on track · K unknown" when K > 0). The 1:1-fallback `convert` is never called by redesigned screens.
   - **`PlanOverviewLoader` + pure `PlanOverviewMetrics.compute(from:)`** for the overview screen, with **two explicitly different input scopes**: budget spending comes from **date-range-scoped transaction fetches** (not `spendingByBudget`, which is hard-coded to each budget's own period; not whole-table live `@Query`), while savings totals come from an **all-time savings-ledger snapshot scoped to non-deleted goal IDs** (lifetime totals can't be date-windowed).
   - **Budgets headline — no double counting, no dishonest denominator, no invalid canonical.** Canonical-total selection considers only **valid** candidates (standing total budgets with `amountLimit > 0` and not needs-attention), ordered by **period priority (monthly → weekly → biweekly → quarterly → yearly) → oldest `createdAt` → lowest UUID** (total order, unit-tested). If a canonical exists: calendar-month all-expense spending vs its monthly-equivalent limit. If only category budgets exist: spending = this-calendar-month spending over the **union** of live tracked categories (each transaction once); limit = sum of monthly-equivalent limits — **but if any two standing category budgets' scopes overlap, the headline degrades** to calendar-month spending + the exact on-track line (no "of Y", no progress bar). If total budgets exist but none is valid: the card renders a needs-attention/setup state without aggregate progress. Monthly-equivalents cover **every** enum case: monthly ×1, weekly ×52/12, biweekly ×26/12, quarterly ÷3, yearly ÷12; custom one-offs excluded.
   - **`BudgetSpendingProjection` — rate-checked per-budget math.** A Plan-scoped projection (spent in budget currency + determinacy) computed with `convertOrNil`. Used by **budget list rows, budget detail stat card, trend chart, and every on-track count**. Existing 1:1-fallback APIs (`spendingByBudgetCurrency` etc.) stay untouched for their existing callers.
   - **`SavingsLedgerCalculator` — one nonisolated ledger authority.** `SavingsGoalReconciler` is `@MainActor` and its `Result` isn't `Sendable`, so background loaders can't legally call it. The ledger math is **extracted into a nonisolated, snapshot-based `SavingsLedgerCalculator`** (value inputs, `Sendable` outputs); the reconciler and the Plan metrics layer both delegate to it — one formula, two callers, no semantic change (a parity test pins reconciler results == calculator results, including its strict live-rate-table rule — see the determinacy bullet).
   - **`SavingsGoalMetrics` — the single savings authority for screens.** One pure struct computed from the `SavingsLedgerCalculator` result + goal fields: saved, remaining, progress, monthly target, completion, behind-status, determinacy. The model's `totalSaved`/`remainingAmount`/`suggestedMonthlyContribution`/`isOnTrack` (which bypass the reconciler) are **not called by any redesigned screen**. An indeterminate goal stays in the Active segment with the partial-data state and no behind/completed claim.
   - **Savings card states, explicitly:** active goals exist → "Saved X of Y" over active goals; zero goals → zero-state CTA card; **all goals completed → distinct "All goals completed" state showing lifetime saved** (never a 0/0 progress bar); indeterminate → partial-data state per the determinacy rule.
   - **Secondary budgets line** "N of M budgets on track" uses exact per-budget periods via `BudgetSpendingProjection`, determinate budgets only — compensating for the month-lens approximation.

4. **Overview screen (`PlanOverviewView`, rewritten in place in `BudgetTabView.swift`).**
   - `NavigationStack` → `ScrollView` on `systemGroupedBackground`, large title **"Plan"** (tab name unchanged). Hero, inline top-3 rows, and insights card all removed.
   - **Budgets summary card** (accent-tinted `PlanCard`, `NavigationLink` → budget list): icon tile, title + localized subtitle, headline + `PlanProgressBar` + % + on-track line from the metrics snapshot.
   - **Savings summary card** (green-tinted `PlanCard`, `NavigationLink` → goals list): headline/progress/counts from the metrics snapshot.
   - **Quick Actions**: two buttons ("New Budget", "New Saving Goal") opening the two form sheets. The overview's toolbar `+` menu is removed; each list screen gets its own `+`.
   - **Empty states**: a feature with no items renders its card in zero-state (subtitle becomes "Create your first…", headline hidden); card tap still opens the (empty) list.
   - Refresh per `PlanRefreshPolicy`.

5. **Budget list (`BudgetListView`, rewritten).**
   - Segmented control **Active | Ended** (mirrors savings' Active | Completed). Active = standing budgets, then upcoming/active one-offs (upcoming labeled "Starts …"); Ended = finished one-offs showing final under/over result.
   - Rich rows in a native `insetGrouped` `List`: category icon tile (or "total" icon), name, period label, spent / limit in the budget's own currency via `BudgetSpendingProjection`, `PlanProgressBar` (category/status color, red when over), %, partial-data state when indeterminate.
   - **Data strategy:** a generation-checked scoped loader (keyed on segment per `PlanRefreshPolicy`) assembles each listed budget's own period range — including ended custom one-offs' historical windows — then **merges overlapping/adjacent ranges (never spanning gaps), fetches each merged interval once, and distributes snapshot rows to budgets in memory** (standing budgets share weeks/months, so naive per-budget fetches would re-read the same rows repeatedly; a collapsed min-to-max span would over-fetch across gaps). Range merging is unit-tested. **Every Plan budget fetch passes `excludeArchivedWallets: false` explicitly** — the shared descriptor's default drops archived-wallet rows before the relevance predicate ever runs, which would contradict the archived-spending-counts rule; an end-to-end loader test with an archived source wallet pins this. Search filters in-memory.
   - **Needs-attention** budgets lose their separate section: the affected row shows an orange warning line (existing `needsAttention` + duplicate-total detection preserved as row state).
   - Filter menu and `PlanBudgetFilter` deleted; `searchable` kept; swipe-to-delete kept but confirms via `confirmationDialog` and **routes through the same transactional save/restore helper and error presentation as the form's delete** (no `try?` path survives in redesigned deletion).
   - Toolbar `+` opens `BudgetFormView` (create mode).

6. **Budget detail (`BudgetDetailView`, rewritten).**
   - `ScrollView` of `PlanCard`s: **hero** (category icon in tinted rounded square, budget name, current-period range) → **stat card**: "Spent X of Y", progress bar, %, plus **Remaining | Left per day** (remaining ÷ days left incl. today). Upcoming one-offs show "Starts in N days"; ended one-offs show final "Under/Over by Z". All amounts via `BudgetSpendingProjection`; indeterminate ⇒ partial-data state, no over/under claims.
   - **Spending Trend card**: cumulative daily spending line/area chart (Swift Charts) over the current period, dashed `RuleMark` at the limit; hidden when the period has no transactions; suppressed for degenerate windows (e.g. single-day one-offs). Own-currency, rate-checked; KHR axis labels use Pro Analytics' compact formatting.
   - **One relevance predicate.** `BudgetCalculator`'s transaction-relevance rules (expense, `deletedAt == nil`, `event == nil`, `!excludeFromReports`, archived wallets **included**, tracked-scope match) extracted into a single shared, tested predicate used by the recent-transactions card, the trend chart, **and** See All.
   - **Recent Transactions card**: latest 5 relevant transactions, rows tap → existing `AddTransactionContainer` edit; **See All** → `FilteredTransactionsDetailView` via an extended `TransactionFilterConfig`: explicit report-exclusion and archived-wallet policy flags **plus an explicit summary currency and opt-in safe-conversion policy applied to everything that converts** — the summary total (today hard-wired to preferred currency via `TransactionProcessor.calculateTotal`'s 1:1 fallback; budget See All displays the budget's currency) **and amount-based sorting** (highest/lowest currently sorts on fallback-converted values); indeterminate rows get stable placement (sorted last, shown unconverted with a hint). All new options default to current behavior for existing call sites.
   - **Data strategy:** period-scoped fetch for the single budget (`excludeArchivedWallets: false`, per the list-loader rule), generation-checked, refresh per `PlanRefreshPolicy` (so "left per day" survives midnight/timezone/foreground transitions).
   - Donut, summary, and insights sections removed. Toolbar **Edit** opens `BudgetFormView` (edit mode) as a sheet.

7. **Savings list (`SavingsGoalListView` + `SavingsGoalRowView`, rewritten).**
   - Segmented **Active | Completed**, driven by `SavingsGoalMetrics` completion. Toolbar `+` → `SavingsGoalFormView`. `searchable` kept (in-memory).
   - Rich rows matching the budget-list recipe: icon tile in goal color, name, target date, saved / target, `PlanProgressBar`, %, "behind" badge — all from `SavingsGoalMetrics`.
   - **Data strategy:** a goal's ledger is unbounded over time, so per-goal metrics are **not** recomputed from relationships during rendering — a cached private-context snapshot loader computes all-time `SavingsGoalMetrics` for all goals once per invalidation (`PlanRefreshPolicy`, keyed on segment), same loader pattern as the budget list.
   - Swipe-to-delete with confirmation, routed through the same transactional save/restore helper as the form's delete. `SavingsGoalSummaryCard` deleted. Goal visuals stay **icon + color** — no photos (future feature).

8. **Savings detail (`SavingsGoalDetailView`, rewritten).**
   - Mirrors budget detail: **hero** (large gradient icon block in goal color, name, target date, completed badge when done) → **stat card**: "Saved X of Y", progress, %, **To Go | Monthly Target** (from `SavingsGoalMetrics`; Monthly Target hidden without a target date) → **Progress card**: cumulative-saved-by-month bar chart → **Recent Contributions card**: latest 5 ledger rows (contribution/withdrawal icons, signed amounts, rate-checked conversion) with **See All** → simple pushed `List` of the full ledger.
   - **Chart ties to the stat card by construction:** opening balance = stamped legacy starting balance (via `startingBalanceCurrencyCode`) + every eligible ledger row before the window; in-window rows accumulate **signed and raw** (withdrawals subtract; `max(0, ·)` applied only per displayed point, never to the running balance); **future-dated rows get an explicit trailing "Upcoming" bucket** so the series total always equals the `SavingsGoalMetrics` saved amount. Chart series computed by a tested pure function from ledger snapshots; ledger fetched via the detail's scoped loader, refresh per `PlanRefreshPolicy`.
   - **Actions**: prominent full-width **Add Money** (goal color, `.glassProminent`) + smaller plain **Withdraw** beneath, both via the existing `SavingsContributionSheet` **upgraded with a locked-goal mode**: opened from a goal detail, the goal is fixed and the transaction form's goal picker is locked/hidden — required because `AddTransactionView`'s picker excludes completed goals and would orphan a preselected completed goal. Contribution and withdrawal on a completed goal are explicitly tested. Completed goals keep both actions (completion is derived).
   - Toolbar **Edit** → `SavingsGoalFormView` sheet.

9. **Unified forms (the add/edit consistency fix).**
   - **`BudgetFormView`** replaces `AddBudgetView` + `EditBudgetView`; **`SavingsGoalFormView`** replaces `AddSavingsGoalView` + `EditSavingsGoalView`. Each is one `Form` with `init(existing: X? = nil)`; **identical sections in both modes**, presented as a sheet from both the `+`/Quick Actions and detail's Edit.
   - **Explicit field-parity matrix** (from the audited current forms):
     - *BudgetFormView*: what-to-budget picker (total/categories + `MultiCategoryPicker`) · amount + currency · suggestion row (**create-only**) · optional name · period type + custom start/end dates · **alert mode picker** (`BudgetAlertMode` — Edit's legacy 50/80/100 toggles are dropped in favor of Add's picker, PR #20's canonical alert model; deprecated `alertAt50` stays unread) · duplicate-total warning (**edit mode excludes the edited budget's own id**, same canonical ordering as the metrics layer) · Delete (**edit-only**).
     - *SavingsGoalFormView*: Quick Start templates (**create-only**) · name · target amount + currency · target date toggle + picker · appearance (icon + color pickers) · **linked wallet picker (both modes** — today it exists only in Add; Edit gains it, no feature silently dropped) · Delete (**edit-only**). No description/priority fields exist in the current forms and none are added.
   - **Currency change is never silent** (both forms): changing currency with a nonzero amount prompts **Convert amount / Keep number / Cancel** (convert via `convertOrNil`; missing rate ⇒ Convert disabled with an explanation). Replaces today's silent re-denomination.
   - **Transactional save path** per form: the form edits **local `@State` copies only**; model fields are written inside the save step. Validate → apply to model (create: insert) → stamp `updatedAt`/`needsSync` → `modelContext.save()` in `do/catch` → post `.dataDidUpdate` **only on success**. **On failure: restore the captured pre-edit values onto the model (create: delete the just-inserted object) so autosave can never persist a half-applied edit**, keep the form open, show the error. Same restore discipline for the delete path (un-tombstone on failed save). The save/restore seam is a small injectable helper so failure paths are unit-testable.
   - **Delete flow fully specified:** red destructive button at the form bottom (edit mode) → `confirmationDialog` → `SoftDeleteService` + explicit save → on success dismiss the form **and pop the underlying detail** (`onDeleted` callback / dismiss chain), so no screen is left showing a tombstoned model.
   - **Suggestion race fixed in the new form:** the create-mode suggestion request carries a generation token; results apply only if the token still matches. Engine scoring untouched.

10. **Wiring & deletion sweep.** `BudgetTabView` remains the tab entry point (ContentView untouched). Before deleting each replaced file (`AddBudgetView`, `EditBudgetView`, `AddSavingsGoalView`, `EditSavingsGoalView`, `SavingsGoalSummaryCard`), grep for external references (Home quick actions, `AppRouter` deep links, onboarding, notifications) and repoint each call site. Xcode filesystem-synchronized groups — no pbxproj edits.

11. **Localization.** All new UI strings use **`.localized` string keys** — the live convention of this codebase (the entire current Plan tab and Pro Analytics ship this way with zero `L10n` constants for their keys). `AGENTS.md`'s "regenerate `String+Localization.swift`" line describes the `L10n`-constant path and is **stale** as a blanket rule (the repo contains no generator script); **this PR amends that AGENTS.md/CLAUDE.md bullet** to document the actual two-path convention: `L10n` constants ⇒ regenerate the generated file; `.localized` keys ⇒ add to both `.strings` + prove EN/KM parity with `Scripts/check_missing_keys.py`. **The script is fixed in this PR to be usable as verification**: repository-relative paths (it currently hardcodes an absolute developer path) and a nonzero exit code when either language is missing keys (it currently always exits 0) — then run as part of this PR's checks. The generated file is never hand-edited. Keys from removed UI are pruned **only when a repo-wide grep proves zero remaining references** (both `.localized` and `L10n.` forms); any `L10n` constant whose key survives pruning stays untouched. All text via `.appFont` (Dynamic Type), dates via Khmer-aware formatting, money stays Western digits.

12. **Verification.**
    - **Unit tests (deterministic, injected date/calendar/rates):** `PlanOverviewMetrics` (canonical-total validity filter + ordering incl. tie-breakers, union path, overlap-degradation, no-valid-total attention state, every period-type monthly-equivalent incl. biweekly, indeterminacy propagation, empty/all-completed savings states) · `BudgetSpendingProjection` (cross-currency rate-checked math, indeterminate exclusion from on-track counts) · hardened `convertOrNil` (zero/negative/NaN/infinite rates on both sides) · end-to-end loader test with an archived source wallet (`excludeArchivedWallets: false` honored) · `SavingsGoalMetrics` (reconciler-derived remaining/monthly-target/behind, legacy starting balance, indeterminate ⇒ no claims) · savings chart series (opening balance with pre-window rows, signed accumulation with per-point flooring, future "Upcoming" bucket tie-out) · shared relevance predicate (archived wallet, excludeFromReports, event, tombstone) · list-loader range assembly (per-budget ranges incl. ended customs, **overlap/adjacency merging without gap-spanning**) · **`SavingsLedgerCalculator` ↔ reconciler parity** (identical results on the same inputs) · **See All safe-conversion policy** (summary currency, indeterminate rows sorted last/unconverted, defaults byte-identical for existing callers) · edit-mode duplicate detection (self-exclusion) · currency-change outcomes · save-failure restore through the shared helper (edit revert, create delete, delete un-tombstone — form and swipe paths).
    - **Manual smoke (simulator, iOS 26.4.1 runtime):** create/edit/delete budget and goal through both entry points; contribution + withdrawal including on a completed goal; segment states (upcoming/active/ended, active/completed); delete-from-detail navigation; empty states; day-rollover/currency-switch refresh; dark mode; Khmer; Dynamic Type XXL spot-check; one Pro Analytics drill-down to confirm `TransactionFilterConfig` defaults unchanged. (Scripted XCUITests stay out of scope — no existing UI-test infra; flows covered manually per this matrix.)
    - Build + full existing test suite green. Then PR per git workflow.

## Key decisions & tradeoffs

1. **Overview = two cards + quick actions only** — hero, inline top-3 rows, and insights card removed. Tradeoff: reaching a specific budget takes one more tap; accepted for simplicity and sample fidelity.
2. **Budgets headline: valid canonical total (total-ordered selection) → union-of-categories → degraded spending-only headline on overlap → attention state when no valid total** — never a dishonest denominator or an unusable headline, by construction.
3. **A real Plan metrics layer, honestly scoped** — loader/compute split per repo concurrency rules; `BudgetSpendingProjection` + `SavingsGoalMetrics` + overview metrics are the only new logic, all pure and unit-tested; existing calculator APIs untouched for existing callers. The one internal refactor: `SavingsGoalReconciler`'s ledger math moves into a nonisolated `SavingsLedgerCalculator` both it and the metrics layer call (parity-tested, semantics unchanged) — required because the reconciler is `@MainActor` and can't serve background loaders.
4. **FX failure is a first-class indeterminate state** — no on-track/behind/completed claims and no `N of M` membership for items with missing rates; redesigned screens never call 1:1-fallback conversion, including the See All summary total (opt-in safe policy on `TransactionFilterConfig`). The guarantee is "rate-checked, never 1:1": hardened `convertOrNil` on the budget side, the reconciler's strict live-rate semantics on the savings side (parity preserved by construction).
5. **Both lists get matching segmented controls** — Budgets: Active | Ended; Savings: Active | Completed. The sample's "Archived" maps to derived states; no archive feature invented. Needs-attention demoted to inline row state.
6. **One `PlanRefreshPolicy` for all five screens** — complete invalidation set (data, rates, preferred currency, language, day/timezone, scene) + segment-keyed loaders; search never refetches.
7. **Detail = hero → stat card → trend chart → recent 5 + See All** — donuts and insights dropped; one chart per screen; See All made exactly consistent via extended filter flags; savings chart ties to the headline via signed accumulation + future bucket.
8. **One shared form per feature, delete inside edit mode** — swipe-to-delete stays as a shortcut but now confirms; delete-from-detail pops the detail; alert UI standardizes on `BudgetAlertMode`; Edit gains the linked-wallet picker rather than silently dropping it.
9. **Currency changes prompt Convert / Keep / Cancel** on both features — replaces silent re-denomination.
10. **Transactional form saves** — local-state editing, apply-at-save, restore-on-failure so autosave can't persist half-applied edits; scoped to the redesigned flows.
11. **Icon + color goal visuals, no photos**; **Add Money prominent + Withdraw small underneath** via locked-goal contribution mode so completed goals keep working.
12. **Flat ProCard-style surfaces with soft tints, not Liquid Glass content cards** — matches Pro Analytics; glass stays in native chrome only. **Keep "Plan" naming.**
13. **`.localized`-only strings for new UI, and the stale AGENTS.md regeneration bullet gets fixed in this PR** — the convention conflict is resolved by correcting the doc to match the codebase's actual two-path practice, not by inventing a generator or silently ignoring the rule.

## Risks / open questions

- **Weekly × 52/12 month-lens** can read slightly "off" against a user's mental month; mitigated by the exact on-track line and the degradation rules. Fallback if testing shows confusion: restrict the headline to monthly budgets only.
- **Reference sweep before deletion**: Home quick actions, `AppRouter` deep links, onboarding, and notification routing may present the old forms; each call site must be repointed, not compiled around.
- **`TransactionFilterConfig` extension** touches a shared screen — all new flags (result-set policies + safe-total policy) default to current behavior; verified by existing suite + the manual Pro Analytics drill-down check.
- **Locked-goal contribution mode** touches `AddTransactionView`'s goal-picker area — confined to "picker hidden/locked when a goal is preselected"; regular entry paths untouched.
- **Invalidation breadth** (day change, timezone, scene-active) adds recompute triggers — all funnel through one generation-checked refresh path, so worst case is a redundant cheap recompute.
- **Ended-one-off history depth**: per-budget ranged fetches keep each query bounded, but a user with many ended one-offs loads several ranges when opening the Ended segment — acceptable (segment is on-demand) and monitored during manual smoke.
- **KHR magnitudes on chart axes** — reuse Pro Analytics' compact axis formatting.

## Out of scope

- Any SwiftData model, Supabase schema, or sync-engine change; goal photos/images; notification/alert behavior changes; widget or iPad-specific layouts; renaming the Plan tab; changes to `BudgetCalculator` period semantics or `BudgetSuggestionEngine` scoring; an app-wide `try?`/error-handling audit (only the new forms get the transactional save path); scripted XCUITest infrastructure; a `String+Localization.swift` generator script (the doc fix documents the existing two-path convention instead); the Home tab's budget/savings surfaces beyond repointing navigation targets; `AddEventLedgerTransactionViewModel` and event screens.
